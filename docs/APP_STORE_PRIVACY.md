# App Privacy — pre-filled nutrition label answers

Apple asks ~30 yes/no/list questions when you fill out the **App Privacy** section in App Store Connect. Below is the complete walkthrough, derived from the locked architecture. Copy each answer into the corresponding ASC field.

**Architectural ground truth:**
- Anonymous-by-default Supabase account on first launch (UUID, no identity)
- Optional Sign in with Apple linkage (Apple-supplied identifier; email if user shares)
- Apple-handled billing (StoreKit), entitlement managed by RevenueCat
- Server-side PHI scrubber rejects patient-identifying inputs before any model call
- Third-party processors enumerated in `/privacy`: Apple, Supabase, RevenueCat, Anthropic, OpenAI, Sentry, PostHog, Vercel
- No advertising SDKs, no IDFA collection, no tracking across apps/websites
- US-only at v1 launch
- Architecturally PHI-free — no name, MRN, DOB, or chart fields

**Used to track you (across apps and websites you don't own):** **NONE.** Skip the tracking section entirely.

## Data Linked to You

When ASC asks "Do you collect this data?" answer **Yes** for the items below; mark each as **Linked to the user's identity** and select the listed purposes. None are used for **Tracking**.

### Contact Info

| Type | Collect? | Linked? | Purposes |
|---|---|---|---|
| **Name** | Yes | Yes | App Functionality |
| **Email Address** | Yes | Yes | App Functionality |
| Phone Number | No | — | — |
| Physical Address | No | — | — |
| Other User Contact Info | No | — | — |

*Notes:*
- *Name:* Optional **display name** in Profile — used for the in-app greeting and Profile header. The user controls whether to enter anything; it can be a nickname, initials, or left blank. We don't ask for legal name.
- *Email:* Only collected if the user opts to sign in with Apple AND chooses to share email. Otherwise the account is fully anonymous. Used for cross-device sync recovery and to receive customer support replies.

### User Content

| Type | Collect? | Linked? | Purposes |
|---|---|---|---|
| Emails or Text Messages | No | — | — |
| Photos or Videos | No | — | — |
| Audio Data | No | — | — |
| Gameplay Content | No | — | — |
| **Customer Support** | Yes | Yes | App Functionality |
| **Other User Content** | Yes | Yes | App Functionality |

*Notes:*
- *Audio Data:* Voice input uses on-device `SFSpeechRecognizer`. Audio never leaves the device — only the transcribed text is sent to the AI co-pilot for processing.
- *Customer Support:* Email content the user sends to `hello@nursemind.app`.
- *Other User Content:* The user's saved bookmarks, profile preferences (role/unit/years), and AI conversation history. Stored in our Supabase Postgres database for cross-device sync. Conversation history is retained for up to 90 days for quota enforcement and abuse review.

### Identifiers

| Type | Collect? | Linked? | Purposes |
|---|---|---|---|
| **User ID** | Yes | Yes | App Functionality |
| Device ID | Yes | **No** | Analytics, App Functionality *(see Data Not Linked below)* |

*Notes:*
- *User ID:* The anonymous UUID generated by Supabase Auth on first launch. If the user opts to sign in with Apple, this UUID is preserved and aliased to the Apple identifier on the backend.

### Purchases

| Type | Collect? | Linked? | Purposes |
|---|---|---|---|
| **Purchase History** | Yes | Yes | App Functionality |

*Notes:*
- We receive subscription entitlement state (active / inactive / trial / monthly / yearly) from Apple via RevenueCat. We do not see the user's payment card, billing address, or Apple ID password. Used solely to gate Pro features.

## Data Not Linked to You

For these, answer **Yes** to collection but **No** to linked-to-user. They're tied to anonymous installation IDs only, not to a user account.

*Notes:*
- Sentry uses a per-install device identifier in crash reports. PostHog uses a per-install distinct ID for product analytics. Neither is tied to the user's NurseMind account or Supabase user record.

| Category | Status |
|---|---|
| Health & Fitness | **Not collected** — no HealthKit integration |

## Tracking

**Q: Do you or your third-party partners use data from this app to track users?**
**A: No.**

We do not link user-or-device data from this app with data from other companies' apps, websites, or offline properties for targeted advertising or measurement. We do not share data with advertising networks. We do not embed the IDFA or any cross-app identifier. PostHog analytics is opt-in and uses an installation-scoped ID with no cross-app identity.

Skip the entire Tracking subsection.

## What changes if you add features later

Track these as future Privacy-label updates if you ship them:

- **Push notifications via APNs** — no new data category (uses anonymous device token)
- **Sign in with Google** (if added in v1.1) — same email handling as Sign in with Apple, no change to label
- **HealthKit integration** (v2+) — would add Health & Fitness category, which is a major label change
- **In-app survey or NPS prompt** — adds User Content → Customer Support if responses are stored
- **Programmatic ads** (we won't) — would add Tracking, change everything

When in doubt, re-read this doc before filling out the App Privacy form for any future release.
